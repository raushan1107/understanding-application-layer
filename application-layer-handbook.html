Back-End Framework
ASP.NET Core Web API
Microsoft · Cross-Platform · Open Source · .NET 6/7/8
What is it?

ASP.NET Core Web API is a server-side framework for building HTTP-based services. It listens for incoming HTTP requests (GET, POST, PUT, DELETE), routes them to the right Controller, processes business logic, and returns structured data — usually JSON.

Think of it as the receptionist + operations engine of your application. Every request from the browser, mobile app, or Blazor UI goes through this layer first.

Analogy A Web API is like a restaurant kitchen. The waiter (browser/Blazor) takes your order (HTTP request) and passes it through the kitchen window. The kitchen (API) validates it, processes it, and returns the cooked meal (JSON/data). You never walk into the kitchen yourself.
Why ASP.NET Core? Key Strengths

Performance: Consistently benchmarked as one of the fastest server frameworks in the world (TechEmpower). Runs on Kestrel, a non-blocking async server.

Built-in DI (Dependency Injection): Services are registered in Program.cs and injected wherever needed — enabling clean, testable code.

Cross-platform: Runs on Windows, Linux, macOS — important for containerised (Docker/Kubernetes) cloud deployments.

Middleware pipeline: Auth, logging, rate-limiting, CORS — each as composable middleware. You control the request lifecycle.

First-class Azure integration: Azure App Service, Azure Functions, Azure AD — ASP.NET Core is the native choice for Microsoft cloud.

Alternatives Comparison
ASP.NET Core Web API CHOSEN
Best fit for .NET ecosystem, Azure-native, Blazor-friendly, world-class performance
Node.js / Express ALT
JS-first teams. Great for microservices but loses type safety and .NET ecosystem benefits
Django / FastAPI ALT
Python ecosystem. FastAPI is excellent for AI/ML backends but not the choice for a .NET project
Spring Boot ALT
Java enterprise. Equivalent feature set but requires JVM, Java devs, and separate toolchain
Where it fits in the architecture

The Web API sits at the core of the back-end. It receives requests from the front-end (Blazor UI / SignalR), validates them, delegates to service classes (business logic layer), which call repositories, which call the database. It is the glue between UI and data.

Architecture Pattern
N-Tier Distributed / Clean Architecture
Layered Design · Separation of Concerns · Dependency Inversion Principle
What is it?

N-Tier Architecture means the application is divided into logical layers (tiers), where each layer has a specific responsibility and only communicates with adjacent layers.

Clean Architecture (Uncle Bob) goes further — it enforces a dependency rule: inner layers know nothing about outer layers. Business logic never imports UI or database code.

Analogy Think of a hospital. The Receptionist (UI/Controller) takes patient details. The Doctor (Service/Business Logic) makes decisions. The Pharmacist (Repository) dispenses medicine. The Stockroom (Database) stores supplies. None of them swap roles — and the Doctor never enters the stockroom directly. Each layer does one job.
The layers — what goes where

Presentation Layer (Controllers / Blazor Pages): Receives HTTP requests. No business logic here — just receives input, calls a service, returns a response.

Application / Service Layer: Business rules live here. Orchestrates use cases. E.g., OrderService.PlaceOrder() validates, calculates, and calls the repository.

Domain Layer (Core): Pure C# entities and interfaces. No external dependencies. This is the heart that never changes even if you swap SQL Server for MongoDB.

Infrastructure / Repository Layer: Implements interfaces from the domain. Talks to EF Core, external APIs, file system. SqlOrderRepository : IOrderRepository.

Data Layer: The actual database (SQL Server, Oracle).

Why this over alternatives?
N-Tier / Clean Arch CHOSEN
Testable, maintainable, swap-in/out database or UI, team scalability — enterprise standard
Monolith (no layers) AVOID
Fast to start but becomes spaghetti code at scale. Business logic leaks into UI. Untestable.
Microservices LATER
Better for large independent teams. Operational complexity (service mesh, distributed tracing) unwarranted for most enterprise apps
CQRS + MediatR EXTENSION
Separates reads and writes — powerful addition to Clean Architecture in complex domains, not a replacement
Architecture placement

The architecture pattern is meta-structural — it doesn't sit in one layer, it governs how all layers relate. Without it, ASP.NET Core becomes a pile of spaghetti. With it, the entire stack becomes predictable, testable, and onboardable.

Communication Protocol
SignalR
Microsoft ASP.NET Core SignalR · WebSockets · Server-Sent Events · Long Polling fallback
What is it?

SignalR is Microsoft's real-time communication library. It creates a persistent, bidirectional connection between server and client. The server can push data to connected clients without the client asking first.

It automatically selects the best transport: WebSockets (preferred), Server-Sent Events, or Long Polling — depending on what the client and server support.

Analogy Regular HTTP is like sending a letter — you post it, wait for a reply. SignalR is like a phone call — both sides can speak at any moment, and either side can hang up or push new information instantly. A stock price ticker, a live dashboard, or a collaborative form all need the "phone call" model.
Critical role: Blazor Server's backbone

In Blazor Server, SignalR is not optional — it IS the Blazor circuit. Every click, keystroke, and state change in a Blazor Server component travels over a SignalR connection to the server, gets processed, and the DOM diff is pushed back to the browser. Blazor Server cannot function without SignalR.

Use Cases in this project

→ Live dashboard metrics refreshing without page reload
→ Notifications pushed from server to logged-in users
→ The Blazor Server UI circuit (all component interactions)
→ Collaborative editing features (if any)

Alternatives
SignalR CHOSEN
Native .NET, automatic transport negotiation, scale-out via Azure SignalR Service, Blazor Server requires it
Raw WebSockets LOW-LEVEL
Maximum control but you manually handle reconnection, heartbeats, fallbacks. SignalR does all this for you
Server-Sent Events ONE-WAY
Server → client only. Good for live feeds but not bidirectional. SignalR uses SSE as fallback anyway
gRPC Streaming MICROSERVICES
Excellent for service-to-service streaming. Not browser-native — needs gRPC-Web proxy for browser clients

Front-End Architecture
Blazor Server-Side (SSR) + HTML / CSS / JS
Razor Components · C# on Server · DOM diff over SignalR · jQuery for legacy interop
What is Blazor Server?

Blazor Server is a Microsoft UI framework that lets you write interactive web UIs in C# — without writing JavaScript for application logic. Components are .razor files combining HTML markup and C# code. The rendering and event handling run on the server; only minimal DOM diffs are pushed to the browser over a SignalR circuit.

Analogy Traditional web: you download a whole app to your phone (React/Angular — WASM/SPA). Blazor Server is like a thin client — the screen (browser) shows output, but all the intelligence runs on the server. Like a display terminal connected to a mainframe. Low download, fast boot, but needs a live connection.
Why Blazor Server over alternatives?
Blazor Server (SSR) CHOSEN
Full .NET stack, C# everywhere, no JS build pipeline, fast initial load, great for enterprise line-of-business apps
Blazor WASM ALT
Runs C# in browser (no server needed for UI). Slow cold start, larger payload, offline capable. Use when offline matters
React / Angular JS TEAMS
Rich ecosystem, JS-native, SPA. Requires JS expertise, separate build, API-first. Not ideal for .NET-centric teams
Razor Pages (MVC) SIMPLER
Classic server-rendered HTML. Simple CRUD works well. No real-time interactivity without SignalR add-ons manually

The key trade-off: Blazor Server requires a permanent server connection per user. For 10,000 concurrent users this needs Azure SignalR Service scale-out. For internal enterprise apps with hundreds of users, it's perfect.

Role of jQuery here

jQuery is the veteran of the JavaScript world. In a Blazor Server project it is not the primary UI driver — Blazor owns that. jQuery is used for legacy interop: calling existing JS plugins, manipulating DOM elements that Blazor doesn't own (PDF viewers, legacy widgets), and quick utility tasks via IJSRuntime.InvokeAsync. It coexists, not competes.

Primary UI Libraries
MudBlazor + Radzen Blazor Controls
Component Libraries for Blazor · Material Design · Data Grids · Charts · Forms
What are they?

MudBlazor is a comprehensive, MIT-licensed Blazor component library following Google's Material Design system. It provides 80+ ready-made components: buttons, dialogs, data tables, date pickers, snackbars, navigation drawers — all in idiomatic Razor syntax.

Radzen Blazor Controls specialises in data-heavy components: advanced DataGrids with sorting/filtering/pagination, Charts, Scheduler, and Tree views. It has a free tier and a commercial tier (Radzen Studio for RAD tooling).

Analogy Building UI without a component library is like constructing furniture from raw timber. MudBlazor is IKEA — polished, pre-designed pieces you assemble quickly. Radzen is the specialist cabinet-maker for your kitchen (complex data tables, charts). You use both because they cover different needs.
MudBlazor — strengths

→ Completely free and open source
→ Consistent Material Design aesthetic out of the box
→ Great theming API — one CSS variable change cascades everywhere
→ Active community, excellent docs
→ MudDataGrid covers most enterprise table needs

Radzen — strengths

→ RadzenDataGrid is among the most feature-complete grids available for Blazor (virtual scroll, frozen columns, row grouping)
→ Built-in Chart components (bar, line, pie, gauge) reducing need for a separate charting library
→ Radzen Studio RAD tool generates Blazor pages from database schemas — huge productivity boost for enterprise data entry apps

Alternatives
MudBlazor + Radzen CHOSEN
Free, complementary — MudBlazor for layout/UX, Radzen for data grids/charts
Telerik UI for Blazor PREMIUM
Excellent but paid. Better PDF/Excel export, 24/7 support. Warranted for large budgets
Syncfusion Blazor PREMIUM
Community license free. Wide component range. Steep learning curve for theming
Pure HTML + CSS AVOID
Maximum control but building grids, modals, date-pickers from scratch is 10× more work with inconsistent results

Responsive Framework
Bootstrap 5.3
CSS Grid · Flexbox · Utility Classes · No jQuery dependency · Dark Mode support
What is it?

Bootstrap is the world's most used open-source CSS framework. It provides a 12-column responsive grid, utility classes for spacing/typography/color, and pre-styled HTML components (navbars, cards, modals, alerts). Version 5 dropped the jQuery dependency — it uses vanilla JS for its interactive components.

Analogy Bootstrap is the urban planning grid of your city. Without it, every developer places buildings (divs) randomly. With it, everyone follows a common grid — the city (web page) looks coherent and every building (component) automatically adjusts its footprint when the city expands (responsive breakpoints).
Bootstrap 5.3 specific — what's new

→ Native dark mode: data-bs-theme="dark" attribute toggles the full colour palette — no custom CSS required
→ Colour mode system: CSS variables throughout, easy to override with brand colours
→ No jQuery: Vanilla JS plugins — aligns with Blazor's own JS isolation
→ New utilities: gap utilities, object-fit, enhanced flexbox helpers

Bootstrap's role alongside MudBlazor

You might ask: "We already have MudBlazor — why Bootstrap too?" Bootstrap handles the page-level layout: the shell, the navbar, the sidebar grid, the container widths. MudBlazor handles component-level UI: buttons, dialogs, data tables inside those containers. They operate at different abstraction levels and coexist cleanly.

Alternatives
Bootstrap 5.3 CHOSEN
Universal skill, massive ecosystem, zero-learning-curve for new devs, dark mode out of box
Tailwind CSS UTILITY-FIRST
More powerful design flexibility, but requires build pipeline (PostCSS/npm). Best when design system is custom
Bulma MINIMAL
CSS only (no JS). Elegant but smaller community, fewer components than Bootstrap
Custom CSS only AVOID
Full control but massive dev time on grid/responsive behaviour that Bootstrap already solved
